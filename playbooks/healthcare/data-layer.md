## 1. Patient Data (환자 데이터)

### 데이터 항목

| 데이터 | 수집 시점 | 저장 위치 | 활용 |
|--------|----------|----------|------|
| 기본 정보 (이름, 연락처) | 회원가입 | 서비스 DB | 본인 식별 |
| 증상/진료과목 | 진료 예약 | 서비스 DB | 의사 매칭, 콘텐츠 추천 |
| 진료 기록 | 진료 완료 | 서비스 DB | 재진료 리마인더 |
| 처방 기록 | 진료 완료 | 서비스 DB | 복약 알림 |
| 약 배송 상태 | 배송 과정 | 물류 시스템 | 실시간 알림 |
| 문의 기록 | 문의 시 | Notion/CRM | 문의 패턴 분석 |
| 만족도 | 진료 후 | 서비스 DB | 서비스 개선, 리뷰 유도 |

### AI 활용 시나리오

**증상 기반 콘텐츠 추천:**
```
환자의 진료 이력 (비염, 알레르기)
    → AI 분석: 관련 건강 콘텐츠 매칭
    → 푸시 알림: "봄철 비염 관리 가이드"
    → 재방문 유도
```

**복약 알림 자동화:**
```
처방 기록 (약 이름, 복용 기간, 복용 횟수)
    → Schedule: 복용 시간에 맞춰 알림
    → 처방 기간 종료 시: 재진료 안내
```

**이탈 예측:**
```
데이터 포인트:
    - 마지막 진료일
    - 진료 빈도 변화
    - 앱 사용 빈도
    - 만족도 점수
    → AI 모델: 이탈 확률 산정
    → 고위험군: 리텐션 캠페인 자동 발동
```

## 2. Partner Data (파트너 데이터)

### 데이터 항목

| 데이터 | 수집 시점 | 저장 위치 | 활용 |
|--------|----------|----------|------|
| 병원 정보 | 제휴 가입 | Notion Leads DB | 파트너 관리 |
| 진료 과목 | 제휴 가입 | Notion Leads DB | 환자 매칭 |
| 소재지 | 제휴 가입 | Notion Leads DB | 지역별 분석 |
| 의사 수 | 제휴 가입 | Notion Leads DB | 규모 파악 |
| 월 진료 건수 | 월별 집계 | 서비스 DB | 성과 추적 |
| 활성 상태 | 실시간 | 서비스 DB | 이탈 감지 |
| 제휴 계약 | 계약 시 | Notion Proposals DB | 계약 관리 |

### AI 활용 시나리오

**리드 스코어링:**
```
신규 파트너 문의 데이터:
    - 병원 규모 (의사 수, 월 진료 건수)
    - 소재지 (수요 높은 지역)
    - 문의 내용 (구체성, 긴급도)
    - 기존 비대면 진료 경험
    → AI 스코어: 1~100점
    → 80+: 영업팀 즉시 배정
    → 50~79: 이메일 시퀀스 + 팔로업
    → 50 미만: 자동 안내 이메일
```

**파트너 이탈 예측:**
```
데이터 포인트:
    - 월 진료 건수 추이 (감소 추세)
    - 로그인 빈도
    - 환자 만족도 추이
    → AI: 이탈 위험도 산정
    → 고위험: 계정 매니저 알림
```

## 3. Marketing Data (마케팅 데이터)

### 데이터 항목

| 데이터 | 수집 시점 | 저장 위치 | 활용 |
|--------|----------|----------|------|
| 유입 채널 | 방문 시 | Analytics | 채널별 ROI |
| 콘텐츠 조회수 | 조회 시 | Analytics | 콘텐츠 성과 |
| 콘텐츠 클릭률 | 클릭 시 | Analytics | 콘텐츠 최적화 |
| 검색 키워드 | 검색 시 | Search Console | SEO 전략 |
| 진료 전환율 | 진료 완료 | 서비스 DB | 퍼널 최적화 |
| 캠페인 성과 | 캠페인 종료 | 광고 플랫폼 | 광고 최적화 |

### AI 활용 시나리오

**콘텐츠 성과 예측:**
```
기존 콘텐츠 데이터:
    - 제목, 키워드, 유형, 플랫폼
    - 조회수, 클릭률, 전환율
    → AI 모델: 새 콘텐츠의 예상 성과 예측
    → 고성과 예상: 광고 예산 배정
    → 저성과 예상: 제목/키워드 수정 제안
```

**전환율 최적화:**
```
퍼널 데이터:
    방문 → 회원가입 → 진료 예약 → 진료 완료 → 재방문
    각 단계 이탈률 분석
    → AI: 이탈 지점별 개선 제안
    → A/B 테스트 자동 설계
```

## Notion DB 스키마 (헬스케어 확장)

기본 Leads DB에 헬스케어 전용 필드를 추가한 형태:

| 필드 | 타입 | 용도 |
|------|------|------|
| Name | title | 병원명 또는 환자명 |
| Email | email | 연락처 |
| Phone | phone_number | 전화번호 |
| Company | text | 병원명 (B2B) |
| Source | select | website, partner-page, app, referral |
| Type | select | b2b (병원), b2c (환자) |
| Status | select | new → contacted → proposal → won/lost |
| Score | number | AI 리드 스코어 (1~100) |
| **Department** | select | 내과, 피부과, 소아과, 정신건강의학과 등 |
| **Region** | text | 소재지 |
| **Doctor Count** | number | 의사 수 |
| **Monthly Cases** | number | 월 진료 건수 |
| Follow-up Date | date | 팔로업 예정일 |
| Proposal | relation | 연결된 제안서 |
| Notes | text | 메모 |

## 데이터 보안 및 규제

헬스케어 데이터는 개인정보보호법 및 의료법 준수 필수.

| 항목 | 요구사항 |
|------|---------|
| 개인정보 | 수집 동의, 암호화 저장, 파기 정책 |
| 의료정보 | 의료법 준수, 진료 기록 보관 의무 |
| 마이데이터 | 보건복지부 인증 (의료 마이데이터) |
| 접근 권한 | 역할별 접근 제한 (RBAC) |
| 로그 | 데이터 접근/수정 이력 기록 |

Notion/Make.com에 저장하는 데이터는 리드 정보 (성명, 연락처, 문의 내용) 수준이므로
의료정보에는 해당하지 않으나, 확장 시 별도 의료정보 시스템(EMR) 연동이 필요합니다.
